- `GET /api/m06a`（同上）

  M06A 用 instr 粗篩 + JVM regex 精篩（不用 Python UDF）——全部沿用原本做法。

### 新增端點（新版 webapp 用）

| 端點 | 方法 | 用途 |
|---|---|---|
| `/api/query` | POST | 查詢頁（query.html），body 帶 `dataset: "M03A"/"M06A"` |
| `/api/stats` | POST | 統計儀表板頁（stats.html），日/週/月流量 |
| `/api/stats/meta` | GET | 回報 M03A 資料涵蓋日期範圍（掃 HDFS 分割目錄，不讀資料） |

| `/`、`/query.html`、`/stats.html`、`/css/*`、`/js/*`、`/data/*` | GET | 直接供應 webapp 靜態檔 |

### `POST /api/query` 行為細節

**M03A**（回傳欄位：`time`、`gantry_id`、`direction`、`vehicle_type`、`traffic_volume`）
- 支援 `gantry_ids`（門架群組展開後的清單）、`direction`（N/S/空=全部）、單一 `vehicle_type`（空=全部）。
- 依查詢跨度自動決定時間粒度（與前端圖表彙整邏輯一致，避免全部門架時回傳量爆炸）：
  - 跨度 > 14 天 → 彙整成每日
  - 跨度 > 2 天，或沒選門架（全部門架）→ 彙整成每小時
  - 其餘 → 原始 5 分鐘
- 單次回傳上限 200,000 列，超過會在回應帶 `truncated: true`。

**M06A**（回傳欄位：`origin_time`、`vehicle_type`、`trip_count`、`avg_travel_time_minutes`、`start_gantry`、`end_gantry`）
- 不回原始旅次列，改回「每 15 分鐘 × 車種」的聚合統計（前端會再依跨度重新彙整成小時/日/月）。
- 判定邏輯維持原本語義：TripInformation 依序經過 A→B 就算一筆。
- **語義變更（重要）**：時間軸與時間範圍過濾改以「通過起始門架 A 的實際時間」為準
  （用 `regexp_extract` 從 TripInformation 抽出），不是舊版的旅次出發時間 `DetectionTime_O`。
  這樣趨勢圖才反映該路段的實際通過量。粗篩仍用 `DetectionTime_O`（下界多留一天）確保不漏資料。
- 旅行時間 = 通過 A 到「最後一次」通過 B 的時間差。

### `POST /api/stats` 行為細節

- body：`{period_type: "daily"/"weekly"/"monthly", start_date, end_date}`（限 366 天內）。
- 一次掃描 M03A（`persist()` 後多次聚合）算出：
  - `summary.total_volume` 總量、`row_count`
  - `hourly_series`（只有 daily 會算）、`daily_series`
  - `vehicle_share`（31/32/41/42/5 占比）
  - `road_ranking` 前 10 名：門架流量用 `webapp/data/gantry_groups_frontend.json` 併成路段群組；
    不在群組定義裡的門架以原始 GantryID 顯示。

### 其他

- webapp 靜態檔目錄預設是 `api_server_3.py` 同層的 `webapp/`，可用環境變數 `WEBAPP_DIR` 覆寫。
  已把 `tk/wulin/webapp/` 複製一份到 `pair2api/webapp/`（之後前端更新記得同步這份）。
- 錯誤回應用 FastAPI 的 `{"detail": ...}`，前端的 `getApiErrorMessage` 讀得到。

## 四、後續修正（同日）：M06A 查不到資料

**症狀**：M06A 查詢沒有回應或回空。查出兩個獨立原因：

1. **Spark driver 記憶體不足**：pod 的 `PYSPARK_SUBMIT_ARGS` 沒設 driver memory，JVM 預設 `Xmx1g`。
   舊 API 只 `collect limit 1000` 不需要 shuffle 所以沒事；新的 M06A 要對整月資料做 groupBy 聚合，
   1g 直接 full GC 卡死（RPC timeout 120s）。已在 `pair2api-deploy.yaml` 改成
   `--master local[*] --driver-memory 8g`（節點 62G、可用 48G，pod 無 memory limit）。
2. **M06A 在 HDFS 上只有 2024-01 ～ 2025-12 的資料**（M03A 有到 2026-05-13），
   查 2026 年的區間會直接回 `row_count: 0`。這是資料涵蓋範圍問題，不是程式問題；
   前端日期選單目前只依 M03A 的 meta 提示範圍，選 M06A 時使用者可能選到沒資料的日期。
   （另外 HDFS 上有今天的 `.retype_tmp`，M06A 目錄似乎正在重整，涵蓋範圍之後可能會變。）

修正後實測：M06A `2025-05-13 08:00–09:00`、01F0099N → 01F0005N，7.6 秒回傳
16 列聚合資料（4 車種 × 4 個 15 分鐘區間，平均旅行時間 ~7.6 分鐘）。

## 五、前端小修（同日，改在 `pair2api/webapp/` 這份）

使用者在 M06A 選了反向的起訖門架（北上起點 + 南下終點），API 正確回 0 筆，
但前端沒有擋也沒有提示，看起來像「查不到資料」。改了兩處：

1. `js/query/api.js`：`validateQuery` 新增檢查——起訖門架方向（GantryID 最後一碼 N/S）不同時，
   直接 alert「M06A 的起始與終點門架方向必須相同」。
2. `js/query/forms.js`：選了起始門架後，終點下拉自動只列同方向門架（日期變更時也會維持過濾）。
3. `js/query/gantries.js`：下拉選單改成「道路 → 方向 → 行車順序（北上里程由大到小、南下由小到大）」
   排列（原本按中文名稱排序，地理上不連貫），選項名稱後面加註「｜北上／｜南下」。
4. `js/query/api.js`：再加一條驗證——同方向但「起訖順序與行車方向相反」（例如北上卻選里程小的當起點）
   時直接提示對調起點與終點，因為這種旅次不存在，查了必是 0 筆。

另外 `/api/query` 已加上查詢 log（dataset、時間範圍、起訖門架、回傳列數、耗時），方便之後除錯。

**注意**：這兩個 js 改動只改在 `pair2api/webapp/`（網站實際供應的目錄），
`~/tk/wulin/webapp/` 原始交付版沒動。請把這兩個改動回報給前端負責人納入他的版本，
不然下次從他那邊重新複製 webapp 時會蓋掉。

## 六、UX 大改版（同日晚間）：階梯式選單 + 地圖選點 + 日期防呆 + 跨度上限

從 log 發現「查不到」的最大宗是**日期超出資料範圍**（有使用者用預設日期查了 2026-07，
M06A 只有 2024-01 ～ 2025-12），其次是門架方向/順序選錯、下拉選單按中文名稱排序造成地理上亂跳。

### 後端（api_server_3.py）
- 新增 `GET /api/query/meta`：回報 M03A / M06A 各自的資料涵蓋範圍（掃 HDFS 分割目錄）。
- `POST /api/query` 加**跨度上限**：M03A 366 天、M06A 92 天，超過直接回 400 與明確訊息
  （範圍太大 Spark 掃不完會逾時，改成快速失敗）。

### 前端（pair2api/webapp/，需同步給前端負責人）
- `query.html`：M06A 表單新增「道路」「方向」欄位。
- `gantries.js`：M06A 改**階梯式選單**——道路 → 方向 → 門架（照行車順序排：北上里程大→小、
  南下小→大，名稱附里程數）。終點選單只列「同路線、起點之後」的門架（結構上排除順序顛倒），
  另以 optgroup 列出其他道路的同方向門架支援**跨國道查詢**。
- `map.js` + `forms.js`：**地圖選點**——M06A 模式下點地圖門架，第一下設起點（自動帶入道路/方向），
  第二下設終點；點到反方向門架視為重選起點；同路線點到起點上游自動對調起訖；
  地圖即時顯示「起／終」預覽標記與說明文字。
- 日期選擇器固定開放 **2021-01-01 ～ 2026-12-31**（使用者要求），實際涵蓋範圍顯示在資料集
  提示文字，送出前驗證：超出涵蓋範圍或跨度超限都會擋下並給明確訊息。

### 驗證
- `/api/query/meta` 回報正確範圍；M06A 6 個月查詢回 400 + 明確訊息；正常 1 小時查詢不受影響。

（曾嘗試「地圖跟著道路/方向只畫單一路線」的地圖改版，應使用者要求已還原，
地圖維持顯示全部門架的原行為，M06A 點選起訖功能保留。）
