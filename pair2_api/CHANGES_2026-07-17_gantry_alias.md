# 2026-07-17 查詢 API 更新：門架新舊編號展開 + M03A 月分區相容

修改檔案：`pair2_api/api_server_3.py`（pod 以 hostPath 掛載 `/opt/wulin/pair2_api` → `/app`，改檔後重啟 pod 即生效，不需重建 image）

## 問題一：門架改編號造成查詢結果被低估

### 現象

部分路段的門架歷年曾改編號（對照表共 23 筆異動、約 11 組），例如：

| 舊編號 | 新編號 | 切換日 |
|---|---|---|
| 01F0155N/S | 01F0153N/S | 2024-04-03 |
| 01F0339N | 01F0340N | 2023-04-26 |
| 01F2483N/S | 01F2472N/S | 2022-04-11 |

資料檔裡的門架編號是「當時」的編號。查詢時間範圍若跨過切換日，
用單一編號過濾只會抓到一半時期的資料：

- **M03A**（車流量）：`GantryID = '01F0340N'` 查 2023-04-20～04-30，
  4/26 之前的流量記在 `01F0339N` 底下，完全不會出現 → 前半段變 0。
- **M06A**（旅次）：`TripInformation` 內記的是旅次當時的門架編號，
  regex 只比對新編號時，切換日之前的旅次全部漏掉 → 旅次數被低估。

### 解法：查詢時自動展開別名群組

資料本身**不動**（歷史資料維持當時的編號），在 API 查詢層處理：

1. **載入對照表**：從 HDFS `/dataset/dim/gantry_summary.csv` 讀
   `門架代碼 / is_active / 舊門架代碼 / 新門架代碼` 欄位，
   用 union-find 把互相參照的編號合併成「別名群組」
   （處理 A→B→C 多次改編號的傳遞關係）。只有真的改過編號的門架會進表。
   程序啟動後第一次查詢時載入、之後快取；讀取失敗自動降級為原樣查詢
   （不擋查詢，下次呼叫重試）。

2. **過濾條件展開**（`expand_gantry_ids()`）：使用者選 `01F0340N`，
   實際過濾條件變成 `IN ('01F0339N','01F0340N')`。套用位置：
   - `GET /api/m03a` 的 `gantry` 參數
   - `GET /api/m06a` 的 `gantry_o`、`gantry_d` 參數
   - `POST /api/query`（webapp 查詢頁）M03A 的 `gantry_ids`、
     M06A 的 `start_gantry_ids` / `end_gantry_ids`
     （M06A 的 `instr` 粗篩與 `rlike` 精篩 regex 都用展開後的清單組出來）

3. **結果歸戶**（`alias_to_queried_map()`，僅 webapp M03A 查詢）：
   展開後撈回的資料會同時有新舊編號兩種 `GantryID`，若原樣回傳，
   前端圖表會在切換日斷成兩條線。因此把舊編號的列改寫回
   「使用者查詢時用的那個編號」，輸出成一條連續序列。
   （舊版 GET 端點屬明細查詢，保留資料原始編號不改寫。）

4. **統計儀表板路段排行**（`/api/stats`）：計算 road_ranking 時，
   查不到群組的門架先透過 `canon` 映射到現行編號再找一次，
   改編號前後的流量會歸在同一路段，不再拆成兩列。

## 問題二：M03A 分區結構遷移的查詢相容

`/dataset/M03A` 正從 `year=/month=/day=` 三層遷移為 `year=/month=` 兩層
（與 M06A 統一）。API 原本寫死 day 層 glob，遷移完成後會全部查空。

- `day_paths()`：同時產生 day 層與 month 層兩種 glob，交給 `existing()`
  留下實際存在的那種——**遷移前、遷移後都能查，不用配合切換時間點改版**。
  month 層少了 day 目錄剪枝，但 M03A 一個月僅約 25MB，影響可忽略。
- `m03a_date_range()`：day 層存在時精確到日；否則退回月份層近似
  （最早月第一天～最晚月最後一天，同 M06A 現行作法）。

## 驗證方式

```bash
# 跨切換日查詢（2023-04-26 切換）：修正前 4/26 前的量為 0，修正後連續
curl -s 'http://<svc>/api/m03a?start=2023-04-20%2000:00:00&end=2023-04-30%2023:59:59&gantry=01F0340N&limit=10'

# webapp 查詢頁同路段跨切換日，圖表應為一條連續序列（gantry_id 統一顯示查詢編號）
# 啟動 log 應出現：[gantry-alias] 載入 /dataset/dim/gantry_summary.csv：22 個編號屬於 11 組新舊對照
```

## 注意事項

- 對照表更新後（例如未來又有門架改編號），**重啟 pair2_api pod** 讓快取重載。
- `gantry_summary.csv` 的 `舊門架代碼`/`新門架代碼` 欄位要成對維護
  （程式用 union-find，單邊有寫也能連起來，但建議兩邊都填）。
- M06A 資料樹目前混有兩種欄位型別（2024–2025 全 string、2026-05 起 int/double），
  跨型別邊界的月份範圍查詢會報 `Parquet column cannot be converted`，
  屬既知問題、與本次修改無關。
