## 二、pair2_api `/api/stats` 重查詢優化（方案 A+B）

**檔案**：`api_server_3.py`
**原版備份**：`api_server_3.py.bak-20260717`（同目錄，還原方式見文末）
**dataset／HDFS／Airflow 管線完全沒動。**

**問題**：儀表板查長區間會拖垮服務。
1. 舊版把過濾後的「原始明細」`df.persist()` 進 8g heap，再跑 5 個聚合 job
   （count／總量／時趨勢／日趨勢／車種／門架）。一個月要 cache 1,400 萬列，
   API 允許的 366 天約 1.8 億列 → heap 撐爆，就是 `unable to create native thread`
   OOM（節點 pids-limit=2048）的火種。
2. `/api/stats`、`/api/query` 是 `async def` 直接跑同步 Spark：查詢期間整個
   event loop 被凍住，**全站**（:80、:28000 都打同一個 pair2_api）所有請求排隊，
   幾個學生同時查就雪崩。

**變更內容**：
1. **（A）單趟聚合**：`stats()` 的 Spark 部分改成一個 job——
   `groupBy(時間桶, GantryID, Direction, VehicleType).agg(sum(Volume), count(1))`
   收回 Python（一個月約 5.3 萬列），總量、日/時趨勢、路段排行、車種佔比、
   row_count 全在 Python 端算。**移除 `df.persist()`**。
   時間桶：daily 週期取到小時（供時趨勢圖），其餘取到日。
   回應 JSON 格式與數值與舊版完全相同（已回歸驗證）。
2. **（B）移出 event loop**：重活搬進 `_stats_impl()`，`stats()` 與 `api_query()`
   改以 `run_in_threadpool()` 執行 → 查詢進行中其他請求照常回應。
3. **併發閘門**：新增 `SPARK_QUERY_SEMAPHORE`（預設 4，env
   `MAX_CONCURRENT_SPARK_QUERIES` 可調）。threadpool 化之後查詢可以並行，
   閘門避免一群人同時按查詢把 8g heap / pids-limit 撐爆，超額的排隊等。
4. 新增 `[api/stats]` 耗時 log（比照 `[api/query]`）。
5. 新 import：`threading`、`collections.defaultdict`、`fastapi.concurrency.run_in_threadpool`。

**驗證（2026-07-17，:80 與 :28000 都測過）**：

| 日 2026-05-13 | 1.2s | 0.6s | rows=491,040 total=16,025,469（一致） |

## 追加（同日晚間）：M06A 地圖路線改用實際旅次路徑

**問題**：查詢頁 M06A 起訖跨不同國道（或起訖是 01H/03A 門架）時，地圖只畫起訖兩點直線。
原因：前端 `getOrderedRouteGroupsBetween()` 只會推算「同路名＋同方向＋里程區間」的門架序列，
跨路直接回空；且里程 regex `^\d{2}F(\d{4})[NS]$` 不認 01H／03A。

**改法**：
1. `api_server_3.py`：`query_m06a_new` 對通過 A→B 的旅次抽樣（`limit(24)`，Spark 提前結束、
   成本遠小於主查詢），從 `TripInformation` 切出「首過 A → 末過 B」的門架序列，取眾數；
   `/api/query` 回應新增 `route_gantry_ids`（M03A 為 null）。抽樣失敗只回 null，不影響主查詢。
   新函式：`route_ids_from_trip()`、`find_m06a_route_ids()`。
2. 前端 `map.js`：`renderM06ARouteMap(query, data)` 優先用 `route_gantry_ids` 座標連線
   （新函式 `getActualRouteLatLngs()`），拿不到才退回原本里程推算 → 再退直線。
3. 前端 `gantries.js`：里程 regex 改 `^\d{2}[FHA](\d{4})[NS]$`，01H／03A 也能推算。
4. 兩份前端（`webapp/`、`pair2_api/webapp/`）都以相同修改同步；兩份已分岔，不可整檔互蓋。

**已知限制**：路徑解析度＝門架間距；經過無門架聯絡道的跳接（如國1泰安→國3快官）兩點間
仍是直線弦——那是資料的真實樣貌。極冷門起訖若抽不到樣本，route 為 null、回退舊畫法。

**驗證**：`01F0467N→05F0309S`（國1→汐五高架→國3→國5）回 15 門架完整路徑；
`01F0980S→03F1022S` 回兩點（中間真的無門架）；M03A 回歸正常。已 rollout restart pair2_api。
