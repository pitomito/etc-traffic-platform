# shared — 跨管線共用工具

`put_to_hdfs.sh` 是唯一的檔案：把本機 staging（`year=/month=/*.csv`）
整批上傳到 HDFS `/raw/`，被 [`pair1_pipeline`](../pair1_pipeline/) 的兩條
獨立流程（歷史回填、每日自動化）共用。

從原本各自為政的兩支上傳腳本抽出、合併成一支的原因：兩條流程的本機
staging 輸出路徑格式完全一致，上傳到 HDFS 的邏輯也完全一致，唯一差異只有
「資料怎麼進到 staging」（歷史回填是解壓 tar.gz、每日自動化是直接下載
單日 CSV）——這部分差異已經在各自的來源腳本裡處理掉了，走到這一步兩邊
輸入格式就統一了，因此上傳邏輯沒有理由重複維護兩份。

## 冪等設計

以「當天 HDFS 上已有的檔案數」對比「該類型單日應有的檔案數」判斷完整度：
完整就跳過，不足才覆蓋重傳（`FORCE=1` 可強制無條件覆蓋）。這讓
`put_to_hdfs.sh` 可以被 Airflow DAG 每天呼叫、被回填流程重跑，都不會
因為「資料已經在那裡了」而出錯或產生重複資料。
