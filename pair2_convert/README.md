# pair2_convert — CSV → Parquet 資料湖轉檔工具

把 `pair1_pipeline` 上傳到 HDFS `/raw/` 的原始 CSV，轉成可供
[`pair2api`](../pair2api/) 直接查詢的分區 Parquet 資料湖 `/dataset/`。
分成「日常自動轉檔」「一次性遷移／重整工具」「投產前的調查腳本」三類。

## 日常自動轉檔（Airflow DAG 呼叫，生產路徑）

- **`convert_batch.py`** —核心轉檔腳本，M03A / M06A 共用一支，靠參數決定
  schema 與輸出粒度。逐日／逐月冪等判斷（raw 不完整就跳過、已轉過就跳過），
  寫入採「先寫暫存目錄 → 驗筆數 → 原子 rename 換入正式路徑」，確保線上
  查詢服務永遠不會讀到寫一半的資料。支援 `--auto`（DAG 用，掃最近窗口）
  與 `--start/--end`（手動補一段歷史）兩種模式。
- **`convert_raw_to_parquet.py`** — 面向手動批次的簡化版：不做冪等分區
  管理，單純把 `/raw` 某段時間的 CSV 轉成月分區 Parquet，用於一次性重建
  某幾個月份。

## 一次性遷移／重整工具（資料湖結構演進時使用）

- **`repartition_m03a.py`** — 把既有的「年/月/日」三層分區重寫成「年/月」
  兩層，依來源大小動態估算輸出檔案數、逐月做筆數校驗、支援斷點續跑
  （state file 記錄已完成的月份，可安全中斷重跑）。
- **`migrate_dataset_to_iceberg.py`** — 評估 Parquet 遷移到 Apache Iceberg
  表格式的一次性搬遷腳本（含 schema 對齊、逐月驗證）。
- **`unify_m06a_schema.py`** — 修正歷史資料因來源批次不同造成的欄位型別
  不一致（例如同一資料集有的月份是 string、有的是正確的 int/double），
  逐月重寫成統一 schema，避免跨月查詢時 Spark 因型別衝突而報錯。
- **`run_migrate.sh` / `convert_all_from_raw.sh` / `put_m06a_new.sh`** —
  上述工具的批次執行 wrapper。**刻意設計成「每一輪處理完就讓整個 JVM
  程序結束、下一輪重新啟動」**：容器化 Spark 環境的 cgroup 對行程/執行緒數
  有上限，長時間單一 JVM 累積執行緒最終會把資源榨乾（`unable to create
  native thread`）；用外部迴圈分批重啟 JVM，讓每一輪都拿到全新的資源額度，
  同時靠 state file 銜接進度，重啟後自動接續未完成的部分。

## 調查腳本（`check_*.py` / `chk_*.py` / `cmp_205.py` / `peek_m06a.py`）

資料遷移過程中用來比對兩批資料（不同來源批次、不同 schema）的差異
（列數、日期範圍、欄位分佈），屬於一次性診斷用途，不是常態執行的工具，
保留是為了說明「發現資料不一致時如何用 Spark 快速定位問題根因」的過程。

## 為什麼要處理型別不一致 / 分區結構演進

真實世界的資料湖很少一開始就有完美一致的 schema 與分區設計——資料量成長、
下游需求改變、不同時期的匯入工具不同，都會讓歷史資料出現落差。這個目錄
展示的不只是「寫轉檔程式」，而是一套遇到資料不一致時「先用 Spark 查證
差異範圍與根因、再設計安全（可驗證、可回滾、可斷點續跑）的修復流程」的
方法，而不是直接覆蓋了事。
